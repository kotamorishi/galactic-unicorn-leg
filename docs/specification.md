# Galactic Unicorn LED表示デバイス 仕様書

## 概要

Pimoroni Galactic Unicorn（Raspberry Pi Pico W搭載）を使用したLED表示デバイス。
ローカルネットワーク内のスマホ・PCからWebブラウザ経由で設定を行い、スケジュールに基づいてメッセージ表示と音声再生を自動実行する。

電源駆動を前提とし、省電力対応は不要。

---

## ハードウェア

| 項目 | 仕様 |
|------|------|
| デバイス | Pimoroni Galactic Unicorn |
| コントローラー | Raspberry Pi Pico W |
| ディスプレイ | 53 x 11 RGB LEDマトリクス |
| オーディオ | I2Sアンプ + 1Wスピーカー |
| 接続 | WiFi（2.4GHz） |
| 電源 | USB給電（常時電源） |

## ソフトウェア構成

| 項目 | 技術 |
|------|------|
| 言語 | MicroPython |
| ファームウェア | Pimoroni カスタムMicroPython |
| Webフレームワーク | microdot |
| 非同期処理 | asyncio |
| 時刻管理 | NTP同期 + 内蔵RTC |

## 機能仕様

### F-01: メッセージ表示

LEDマトリクスにテキストメッセージを表示する。

| 項目 | 仕様 |
|------|------|
| 表示方式 | 固定表示 / スクロール表示（選択可能） |
| スクロール方向 | 右から左 |
| スクロール速度 | 設定可能（低速・中速・高速の3段階） |
| 文字色 | RGB指定（Web UIから選択） |
| フォント | bitmap6 / bitmap8（選択可能） |
| 最大文字数 | 128文字（暫定。RAM制約により調整の可能性あり） |
| 繰り返し | スクロール表示時、ループ再生 |

### F-02: スケジュール表示

メッセージの表示タイミングを時間範囲で制御する。

| 項目 | 仕様 |
|------|------|
| 時間範囲指定 | 開始時刻〜終了時刻（HH:MM単位） |
| 複数スケジュール | 1つのメッセージに対して複数の時間範囲を設定可能 |
| 曜日指定 | 各スケジュールに対して曜日を選択可能（月〜日） |
| スケジュール外の動作 | 表示OFF（LEDマトリクス消灯）または待機画面表示 |
| 時刻管理 | 起動時にNTPで同期。以降定期的に再同期（1時間ごと） |
| タイムゾーン | JST（UTC+9）固定（設定変更は将来対応） |

### F-03: 音声再生

メッセージ表示のタイミングに合わせて音を再生する。

| 項目 | 仕様 |
|------|------|
| 再生タイミング | スケジュール開始時（メッセージ表示と同時に再生） |
| 音声ソース | プリセットから選択（ユーザーが音声を追加する機能はなし） |
| プリセット数 | 20種類 |
| 音量 | Web UIから設定可能（0〜100%） |
| 再生回数 | 1回再生（スケジュール開始時のみ） |

### F-04: WiFi初期設定（キャプティブポータル）

初回起動時またはWiFi未設定時に、デバイス自身がアクセスポイント（AP）を起動し、スマホ等からWiFi接続情報を設定する。

| 項目 | 仕様 |
|------|------|
| APモード SSID | `GalacticUnicorn-Setup` |
| APモード パスワード | なし（オープン）またはデバイス固有の初期パスワード |
| APモード IPアドレス | 192.168.4.1（MicroPython APデフォルト） |
| 設定画面 | キャプティブポータル方式。AP接続後ブラウザで自動的に設定画面を表示 |
| 保存先 | Pico W内蔵フラッシュ（`wifi_config.json`） |
| 設定後の動作 | 保存完了後、自動的にAPを停止し設定されたWiFiに接続 |

#### WiFi設定画面

- 周辺のSSID一覧を表示（スキャン結果）
- SSID選択または手動入力
- パスワード入力
- 「接続」ボタン → 接続テスト → 成功時に保存して再起動

#### WiFi設定の再設定

| 方法 | 説明 |
|------|------|
| 物理ボタン | デバイスの特定ボタン（例: AとDを同時長押し5秒）でAPモードを再起動 |
| Web UI | 通常運用時のシステム設定画面からもWiFi設定を変更可能 |

#### 動作判定ロジック

```
起動時: wifi_config.json が存在するか？
  ├── NO  → APモード起動（初期設定）
  └── YES → 保存済みSSIDに接続を試行
               ├── 成功 → 通常運用モードへ
               └── 失敗（30秒タイムアウト）→ APモード起動（再設定）
```

| 項目 | 仕様 |
|------|------|
| プロトコル | HTTP（HTTPS非対応） |
| ポート | 80 |
| 対応ブラウザ | モダンブラウザ（Chrome, Safari, Firefox） |
| レスポンシブ | スマホ・PC両対応 |
| 認証 | なし（ローカルネットワーク内での使用を前提） |

#### Web UI 画面構成

**1. メッセージ設定画面**

- メッセージテキスト入力
- 表示方式選択（固定 / スクロール）
- スクロール速度選択（低速 / 中速 / 高速）
- 文字色選択（カラーピッカー）
- フォント選択

**2. スケジュール設定画面**

- 時間範囲の追加・編集・削除（開始時刻〜終了時刻）
- 曜日選択（チェックボックス）
- スケジュール一覧表示
- 有効/無効トグル

**3. 音声設定画面**

- プリセット音の選択（ドロップダウン + 試聴ボタン）
- 音量スライダー
- 音声ON/OFFトグル

**4. システム設定画面**

- 現在時刻表示

- 現在のアプリバージョン表示
- OTA更新（手動で即時チェック + 更新ボタン）

### F-06: OTA自動更新

GitHubリポジトリの最新コードを定期的にチェックし、更新があればアプリの `.py` ファイルを自動更新する。ファームウェア（UF2）は更新対象外。

| 項目 | 仕様 |
|------|------|
| 更新対象 | `/app/` 配下の `.py` ファイルのみ |
| 更新対象外 | ファームウェア（UF2）、設定ファイル（`wifi_config.json`, `app_config.json`） |
| チェック頻度 | 1日1回（デフォルト午前3:00） |
| チェック方法 | GitHub API でリポジトリの最新コミットハッシュまたはリリースタグを取得し、ローカル保存のバージョンと比較 |
| ダウンロード | `raw.githubusercontent.com` から `.py` ファイルを1つずつ取得・上書き |
| 更新後の動作 | バージョン情報をフラッシュに保存し、自動再起動 |
| 手動更新 | Web UIのシステム設定画面から即時チェック・更新を実行可能 |
| 更新中の表示 | LEDマトリクスに「Updating...」を表示 |
| メモリ確保 | OTA実行時はディスプレイ更新・スケジュール監視を一時停止し、RAM を確保した上でHTTPS通信を行う |
| マニフェスト | リポジトリに `manifest.json`（更新対象ファイル一覧）を配置。OTA時はまずこれを取得し、差分更新する |

#### バージョン管理

| 項目 | 仕様 |
|------|------|
| バージョン識別 | GitHubのコミットハッシュ（短縮7桁） |
| 保存先 | `version.json`（フラッシュ） |
| リポジトリ設定 | `ota_config.json` にリポジトリURL・ブランチ名を記載 |

#### `ota_config.json` 構造

```json
{
  "repo": "username/galactic-unicorn-leg",
  "branch": "main",
  "app_path": "src/",
  "check_hour": 3
}
```

#### `manifest.json` 構造（リポジトリに配置）

OTA更新対象のファイル一覧。リポジトリの `src/manifest.json` に配置する。

```json
{
  "version": "a1b2c3d",
  "files": [
    "main.py",
    "boot.py",
    "config/config_manager.py",
    "display/renderer.py",
    "audio/presets.py",
    "audio/player.py",
    "web/server.py",
    "web/routes.py",
    "web/templates.py",
    "scheduler/scheduler.py",
    "ota/updater.py",
    "lib/microdot.py"
  ]
}
```

#### OTA更新手順（詳細）

1. ディスプレイ更新・スケジュール監視を停止し `gc.collect()` でRAMを確保
2. `manifest.json` をHTTPSで取得
3. ローカルの `version.json` と比較。同じならここで終了、通常動作に復帰
4. 差分がある場合、`manifest.json` の `files` を1つずつダウンロード
5. 各ファイルは `.tmp` に書き込み → `os.rename()` でアトミックに上書き
6. 全ファイル完了後、`version.json` を更新
7. 自動再起動

#### 更新失敗時の対策

| ケース | 対応 |
|--------|------|
| GitHub API接続失敗 | スキップして次回再試行。通常動作を継続 |
| ファイルダウンロード失敗 | 途中で中断し、次回再試行。部分更新されたファイルが残る可能性あるが、次回の更新で修復される |
| 更新後に起動不能 | 物理ボタン長押しでAPモード起動 → Web UIから手動復旧、またはUSB経由で再書き込み |

## システム動作フロー

### 起動シーケンス

```
電源ON
  ↓
wifi_config.json 存在チェック
  ├── なし → APモード起動（キャプティブポータル）
  │            ↓
  │          スマホ等からSSID/Password設定
  │            ↓
  │          wifi_config.json に保存 → 再起動
  │
  └── あり → 保存済みSSIDに接続試行
               ├── 失敗（30秒タイムアウト）→ APモード起動
               └── 成功 ↓
                        NTP時刻同期 → RTC設定
                          ↓
                        app_config.json 読み込み
                          ↓
                        Webサーバー起動（ポート80）
                          ↓
                        メインループ開始
                          ├── スケジュール監視（毎分チェック）
                          ├── Webリクエスト処理
                          ├── NTP再同期（1時間ごと）
                          ├── WiFi接続監視（30秒ごと）
                          └── OTA更新チェック（1日1回）
```

### OTA更新フロー

```
1日1回（例: 午前3時）: GitHub APIに最新バージョンを問い合わせ
  ↓
現在のバージョンと比較（コミットハッシュまたはタグ）
  ├── 同じ → 何もしない
  └── 異なる → 更新開始
                 ↓
               .pyファイルを1つずつダウンロード → フラッシュに上書き
                 ↓
               バージョン情報を更新
                 ↓
               自動再起動 → 新しいコードで起動
```

### スケジュール実行フロー

```
毎分: 現在時刻をスケジュールと照合
  ↓
一致するスケジュールあり？
  ├── YES → メッセージ表示開始
  │         スケジュール開始時刻ちょうどなら音声再生
  └── NO  → 表示OFF or 待機画面
```

## 制約事項

| 項目 | 内容 |
|------|------|
| 解像度 | 53x11ピクセル。一度に表示できる文字数は6〜8文字程度 |
| RAM | 約192KB。Web UI用HTMLは最小限に抑える必要あり |
| Flash | 2MB。ファームウェア + コード + 設定データで共有 |
| 同時接続 | Webサーバーへの同時接続数は1〜2程度を想定 |
| WiFi | 2.4GHzのみ。5GHz非対応 |
| セキュリティ | HTTP通信のため暗号化なし。ローカルネットワーク内での利用を前提 |

## WiFi接続管理

### 自動再接続

ルーター再起動やWiFi不安定による接続切断に対応する。

| 項目 | 仕様 |
|------|------|
| 接続監視 | 30秒ごとに `wlan.isconnected()` で確認 |
| 再接続試行 | 切断検出時、保存済みSSIDへ即座に再接続を試行 |
| リトライ | 最大5回。失敗ごとに待機時間を増加（5秒 → 10秒 → 20秒 → 40秒 → 60秒） |
| リトライ上限超過 | 5分間待機後に再度リトライサイクルを開始。APモードには遷移しない |
| 切断中の動作 | スケジュール表示は継続（RTC基準）。Webサーバーは利用不可。OTAはスキップ |
| 再接続後 | NTP再同期を即時実行。Webサーバーは自動復旧 |

## テスト戦略

### 方針

実機テストの回数を最小限に抑えるため、ハードウェア依存部分を抽象化し、PC上でのテストを最大化する。

### テスト層

| 層 | 実行環境 | 対象 | フレームワーク |
|----|---------|------|--------------|
| Unit | PC (CPython) | ビジネスロジック（スケジューラ、設定管理、OTAロジック） | pytest |
| Web UI | PC (CPython) | HTTPルーティング、API応答、HTMLレンダリング | pytest + httpx（microdotはCPythonでも動作） |
| Integration | PC (CPython) | モジュール間連携（スケジューラ→表示→音声の呼び出しチェーン） | pytest + モック |
| Device | 実機 | ハードウェア操作（LED表示、音声再生、WiFi、ボタン） | 手動テスト + テストスクリプト |

### ハードウェア抽象化

PC上テストのため、ハードウェア依存部分をインターフェース化する。

```
src/
├── hal/                        # Hardware Abstraction Layer
│   ├── __init__.py
│   ├── interfaces.py           # 抽象インターフェース定義
│   ├── real.py                 # 実機実装（GalacticUnicorn, PicoGraphics）
│   └── mock.py                 # PCテスト用モック実装
```

```python
# hal/interfaces.py（概念）
class DisplayInterface:
    def draw_text(self, text, x, y, color, font): ...
    def clear(self): ...
    def update(self): ...

class AudioInterface:
    def play_preset(self, preset_id, volume): ...
    def stop(self): ...

class NetworkInterface:
    def is_connected(self): ...
    def get_ip(self): ...
```

### PC上テスト対象（実機不要）

| モジュール | テスト内容 |
|-----------|-----------|
| **config/** | JSON読み書き、バリデーション、デフォルト値、破損ファイル復旧 |
| **scheduler/** | 時間範囲判定、曜日フィルタ、複数スケジュールの優先度、境界値（23:59→00:00） |
| **web/** | 全APIエンドポイントの正常系・異常系、入力バリデーション、HTMLレスポンス |
| **ota/** | バージョン比較、マニフェスト解析、ダウンロードロジック（HTTPモック） |
| **audio/presets.py** | プリセットデータの構造検証（必須キーの存在、値の範囲チェック） |

### 実機テスト対象（PC上で代替不可）

| テスト項目 | 確認内容 |
|-----------|---------|
| LED表示 | 文字が正しく描画されるか、色・フォント・スクロールの視認確認 |
| 音声再生 | 各プリセットが正しく鳴るか、音量調整が機能するか |
| WiFi AP mode | キャプティブポータルが表示されるか、設定が保存されるか |
| WiFi STA mode | 接続・再接続・NTP同期が動作するか |
| OTA（E2E） | GitHub経由で実際にファイルが更新・再起動されるか |
| ボタン操作 | 物理ボタン長押しでAPモード遷移するか |
| 長時間稼働 | 24時間以上の連続動作でメモリリーク・ハングが発生しないか |
