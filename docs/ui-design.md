# Web UI Design Document

## Current Problems

- 5ページに分かれているが、ユーザーの実際の操作フローと合っていない
- 「メッセージ」と「スケジュール」が別ページだが、これは1つの概念（「いつ」「何を」表示するか）
- 「サウンド」ページが独立しているが、音はスケジュールに紐づく設定
- 設定項目が多すぎる。ユーザーが本当に頻繁に変えるものと、一度設定すれば触らないものが混在
- システム情報が多すぎる。ユーザーが知りたいのは「動いているか」だけ

## ユーザーの行動分析

### 誰が使うか

自宅やオフィスでLED表示板を使う一般ユーザー。技術者ではない可能性が高い。
スマホから操作することが多い。

### 実際のユースケース

1. **初回セットアップ**: WiFi接続 → メッセージ設定 → 完了（1回だけ）
2. **メッセージ変更**: 表示テキストを変える（最も頻繁）
3. **スケジュール調整**: 表示時間帯を変える（たまに）
4. **確認**: 今どんな設定になっているか見る（たまに）
5. **システム管理**: 輝度変更、WiFi変更、再起動（稀）

### ユーザーにとって有意義な情報

| 情報 | 重要度 | 理由 |
|------|--------|------|
| 現在のメッセージテキスト | 高 | 一番変えたいもの |
| 現在の表示状態（ON/OFF） | 高 | 今動いているか知りたい |
| 次のスケジュール開始時刻 | 高 | いつ表示されるか知りたい |
| 文字色 | 中 | たまに変えたい |
| スケジュール（時間・曜日） | 中 | 設定済みなら頻繁には変えない |
| 音のON/OFF | 中 | 状況に応じて切りたい |
| 輝度 | 低 | 一度設定すれば触らない |
| フォント | 低 | ほぼ変えない |
| スクロール速度 | 低 | ほぼ変えない |
| WiFi状態/IP | 低 | 問題が起きたときだけ |
| バージョン | 低 | OTA時のみ |

## 新しいUI設計

### 設計原則

1. **1画面で完結**: 最も使う操作はトップページで完結させる
2. **段階的開示**: 詳細設定は折りたたんで隠す。必要なときだけ展開
3. **状態が見える**: 今の設定が一目でわかる
4. **ページ遷移は最小限**: スマホで何度もページ移動させない

### ページ構成（2ページのみ）

#### Page 1: メイン画面 `/`

ユーザーが最も使う操作をすべてここに集約する。

```
┌─────────────────────────────────┐
│  Galactic Unicorn               │
│                                 │
│  ┌─ Now ──────────────────────┐ │
│  │ "Hello World!"             │ │
│  │ ● Displaying (until 09:00) │ │
│  │ or                         │ │
│  │ ○ Off (next: 08:00 Mon)    │ │
│  └────────────────────────────┘ │
│                                 │
│  ┌─ Message ──────────────────┐ │
│  │ [Hello World!          ]   │ │
│  │ [Save]                     │ │
│  │                            │ │
│  │ ▶ Options (color, font...) │ │  ← 折りたたみ。タップで展開
│  │   Color: [■]               │ │
│  │   Mode:  [Scroll ▼]       │ │
│  │   Speed: [Medium ▼]       │ │
│  │   Font:  [Normal ▼]       │ │
│  └────────────────────────────┘ │
│                                 │
│  ┌─ Schedule ─────────────────┐ │
│  │ ☑ 08:00 - 09:00  Mon-Fri  │ │  ← 既存スケジュール一覧
│  │   🔔 Rising Chime          │ │    音の設定もここに表示
│  │ ☐ 18:00 - 19:30  Sat-Sun  │ │
│  │                            │ │
│  │ [+ Add] [Save]             │ │
│  └────────────────────────────┘ │
│                                 │
│  ┌─ Quick Settings ───────────┐ │
│  │ Brightness  ────●────  65% │ │
│  │ Volume      ──●──────  30% │ │
│  └────────────────────────────┘ │
│                                 │
│  ⚙ Device Settings             │  ← リンク。Page 2へ
│                                 │
└─────────────────────────────────┘
```

**セクション詳細:**

**Now（ステータス）**
- 現在表示中のメッセージと状態を表示
- 表示中なら「Displaying (until HH:MM)」
- 非表示なら「Off — next: HH:MM Day」
- ユーザーが最初に見て「動いてるな」と安心できる

**Message（メッセージ）**
- テキスト入力 + Saveボタンをメインに大きく配置
- 色・フォント・モード・速度は「Options」として折りたたみ
- テキストを変えるだけなら1タップ + 入力 + Save で完了

**Schedule（スケジュール）**
- 各スケジュールをコンパクトな1行で表示
- 有効/無効のチェックボックス
- 音のプリセット名も同じ行に表示（別ページ不要）
- 「+ Add」でインライン追加
- スケジュール行タップで展開（時間・曜日・音の詳細を編集）

**Quick Settings**
- 輝度と音量のスライダーだけ
- 即座に反映、保存操作不要

### 旧UI→新UIの変更まとめ

| 旧 | 新 | 理由 |
|----|-----|------|
| 5ページ（Home, Message, Schedule, Sound, System） | 2ページ（メイン, デバイス設定） + 初期設定 | ページ遷移を最小化 |
| ナビゲーションバー（5タブ） | 不要。1ページに集約 | タブを選ぶ手間がなくなる |
| メッセージとスケジュールが別ページ | 同じページ内のセクション | 「何を」「いつ」は1つの概念 |
| サウンド設定が独立ページ | スケジュール行の中に統合 | 音はスケジュールに紐づく |
| サウンドプレビューが別ページ | スケジュール編集の展開内にPlayボタン | コンテキストを離れない |
| 現在の状態が見えない | Nowセクションで一目でわかる | 安心感 |
| 全設定が常時表示 | 折りたたみで段階的開示 | 画面がスッキリ、迷わない |
| 輝度がシステムページ | メインページのQuick Settings | よく使うものは手元に |

### API変更

既存APIはそのまま活用可能。追加で以下が必要:

| Endpoint | Method | 用途 |
|----------|--------|------|
| `/api/status` | GET | 現在の表示状態（active/inactive、現在のメッセージ、次のスケジュール） |

### 実装の注意点

- 折りたたみはCSS `details/summary` を使用（JS不要、軽量）
- スケジュール行の展開・編集もCSS+最小限のJSで実装
- 1ページあたりのHTML量は増えるが、ページ遷移が減るのでトータルの通信量は減る
- RAM制約: メインページのHTMLが大きくなりすぎないよう、スケジュールが多い場合はコンパクトに
